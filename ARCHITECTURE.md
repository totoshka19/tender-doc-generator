# Записка по архитектуре

## Общая схема потока данных

```
data/incoming/*.docx
        │
        ▼ (опционально)
  extractor.py  ──────────────────────────────────┐
        │                                          │
        │  tender_extracted.json                   │
        │  (заготовка, требует ручного             │
        │   дополнения null-полей)                 │
        ▼                                          │
data/tender.json ◄─────────────────────────────────┘
data/company_profile.json
data/calc.json
        │
        ▼
  validator.py
  (проверка реквизитов и арифметики)
        │
        │  ошибки → остановка
        │  ОК → продолжение
        ▼
  engine.py
  load_data() → единый контекст {profile, tender, calc, system}
        │
        │  для каждого *.yaml из mappings/:
        ▼
  engine.generate_document()
        │
        ├── fill_document()         — простые замены плейсхолдеров
        └── fill_items_table()      — итерация по позициям (только doc 03)
        │
        ▼
output/*.docx
```

## Детерминированная логика vs LLM

Все подстановки в проекте детерминированы: каждый плейсхолдер имеет
явно прописанный путь к источнику данных в YAML-маппинге.

**Почему не LLM для типовых полей:**

- LLM может «угадать» значение, которого нет в данных — это критичная ошибка
  в юридическом документе (неверный ИНН, сумма, реквизиты).
- Детерминированный маппинг даёт гарантию: если данных нет — поле пустое,
  а не выдуманное.
- Детерминированный код воспроизводим: одни и те же данные всегда дают
  один и тот же документ.

LLM допустим только для извлечения полей из неструктурированных входящих
документов (extractor.py), но и там использован детерминированный парсинг
по ключевым словам, поскольку структура входящего DOCX фиксирована.

## Решение проблемы разбитых runs

В формате DOCX текст параграфа хранится в виде runs — фрагментов с
независимым форматированием. Один плейсхолдер `[ИНН]` может быть разбит
на несколько runs: `[И`, `НН`, `]` — например, если часть текста была
скопирована из другого источника или отформатирована по-другому.

Простой поиск по `run.text` в таком случае не найдёт плейсхолдер.

**Алгоритм `replace_in_paragraph()`:**

1. Склеить текст всех runs параграфа в одну строку
2. Проверить наличие плейсхолдера в склеенной строке
3. Если найден — записать заменённый текст в первый run
4. Все остальные runs обнулить (`run.text = ""`)

Форматирование первого run (шрифт, размер, жирность) при этом сохраняется,
остальные runs становятся пустыми строками и визуально исчезают.

## Расширяемость: новый документ = шаблон + YAML

Движок `engine.py` не знает ничего о конкретных документах. Он умеет:
- резолвить пути в данных (`profile.company.inn`)
- заменять плейсхолдеры в параграфах и ячейках
- итерироваться по позициям таблицы

«Знание» о том, какой плейсхолдер откуда берётся — целиком в YAML-маппинге.

**Чтобы добавить четвёртый документ** (например, Референц-лист):
1. Положить шаблон в `templates/04_Референц_лист_шаблон.docx`
2. Создать `mappings/04_ref.yaml` с той же структурой: `template`, `output`, `replacements`
3. Запустить `main.py` — документ будет сгенерирован в числе остальных

Ни `engine.py`, ни `main.py`, ни `validator.py` изменений не требуют.

## Валидатор: что проверяет и почему стоит до генерации

**Что проверяется:**

| Поле | Проверка |
|---|---|
| ИНН | 10 цифр, контрольная сумма по алгоритму ФНС |
| КПП | ровно 9 цифр |
| БИК | 9 цифр, начинается с `04` |
| ОГРН | 13 или 15 цифр |
| `bid_deadline` | дата не в прошлом |
| `offer_validity_days` | положительное целое число |
| Суммы в `calc.json` | `subtotal = Σ line_total`, `vat = subtotal × rate`, `total = subtotal + vat` |

**Почему до генерации, а не после:**

Генерация создаёт файлы. Если ошибка обнаружена после — пользователь
получает частично заполненный или формально неверный документ, который
выглядит готовым. Это хуже, чем явный отказ до начала работы.

Валидатор разделяет ошибки (блокируют генерацию) и предупреждения
(выводятся, но не останавливают процесс) — это позволяет работать
с неполными данными там, где это допустимо.
